# 端到端测试报告

> 测试日期：2026-06-18　|　环境：本地 macOS（非 K8s）　|　执行人：zp

## 1. 测试目标

在本机不依赖 K8s 的前提下，拉起完整链路（gateway → docservice / aiservice → etcd / Qdrant / 本地 Embedding / DeepSeek），
验证 RAG 系统的「文档摄入 → 语义检索 → LLM 生成」端到端闭环是否可用。

## 3. 测试过程中发现的问题

| # | 问题 | 现象 | 根因 | 处理 | 严重度 |
|---|------|------|------|------|--------|
| P1 | Docker 守护进程未运行 | `Cannot connect to the Docker daemon` | OrbStack 未启动 | `open -a OrbStack` 后等待就绪 | 低（前置条件） |
| P2 | **8081 端口冲突** | aiservice 启动 `panic: bind: address already in use` | 本机另一项目容器 `yundun_nginx`（8081→80）长期占用 8081 | 将 aiservice 临时改到 `:8082`；gateway 走 etcd `ai.rpc` 服务发现，换端口对其透明 | 中 |
| P3 | DeepSeek key 不在 shell 环境 | `DEEPSEEK_API_KEY` 未设置 | key 仅存于 gitignore 的 `一些配置.md`，新 shell 不会自动加载 | 启动 aiservice 前从该文件注入 env | 低（设计如此，key 不入库） |
| P4 | 查询被标记 slowcall | ai 日志出现 `slowcall ... duration 882~1156ms` | go-zero 默认慢调用阈值偏低，而单次查询含 Embedding+检索+LLM 往返本就 ~1s | 仅告警，功能正常；可调阈值或忽略 | 极低（观察项） |
| P5 | 无相关性阈值过滤 | 无关问题仍返回 2 条 sources | Top-K 检索总返回最近邻，不论距离远近 | 当前由 LLM 兜底拒答（见 T5）；如需更严可加距离阈值过滤 | 极低（改进建议） |

### 关键验证点
- **摄入链路**：上传 → 分块 → bge-m3 向量化 → Qdrant upsert，点数与维度正确。
- **检索链路**：基于语义相似度排序，T4 正确把相关文档排在首位（非按上传顺序）。
- **生成链路**：DeepSeek 正常返回，答案 grounded 于检索内容。
- **防幻觉**：T5 知识库外问题，模型遵循 Prompt 约束拒答，未编造。

## 5. 结论

**端到端闭环验证通过 ✅**　—— 此前 progress.md 中 `partial / 三服务整链路待跑` 的项现已实测打通。

5 个用例全部 pass。发现的 5 个问题中：
- P1/P3 为前置条件（启动 Docker、注入 key），非缺陷；
- **P2（8081 端口冲突）是唯一需要决策的项**：是否把仓库默认端口改掉，避免本机每次都撞 `yundun_nginx`；
- P4/P5 为观察项 / 可选改进，不影响功能。

## 6. 处置与改进建议

1. **aiservice 默认端口（已处置）**：决定将仓库默认端口由 8081 固化为 **8082**，避免本机长期与 `yundun_nginx` 冲突。已同步更新 `aiservice/etc/ai.yaml`、`deploy/k8s/configmap.yaml`、`deploy/k8s/aiservice.yaml`、README 与本地开发测试文档。对 K8s 无影响（gateway 走 etcd `ai.rpc` 服务发现，不依赖固定端口）。
2. （可选）为检索增加相似度距离阈值，低于阈值直接返回「未找到相关内容」，减少对 LLM 兜底的依赖。
3. （可选）调整 go-zero 慢调用阈值，避免正常 ~1s 的 RAG 查询被记为 slowcall。
