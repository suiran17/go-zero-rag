# 本地开发测试指南

不依赖 K8s，在本机把整套 RAG 系统跑起来并验证端到端问答闭环。

## 架构与依赖

```
client ──HTTP──> gateway(:8888) ──RPC──> docservice(:8080)  文档摄入
                                  └─RPC──> aiservice(:8082)   RAG 问答
                                            │
   etcd(:2379) 服务注册发现 ─────────────────┤
   Qdrant(:6333) 向量库 ────────────────────┤
   Embedding(:1234, bge-m3) 文本向量化 ──────┤
   DeepSeek(api.deepseek.com) LLM 生成 ──────┘
```

本地运行依赖 4 样东西：

| 依赖 | 地址 | 用途 | 来源 |
|------|------|------|------|
| **etcd** | `127.0.0.1:2379` | 服务注册发现（gateway 找 rpc 服务） | docker-compose ✅ |
| **Qdrant** | `127.0.0.1:6333` | 向量库（存/检索文档向量） | docker-compose ✅ |
| **Embedding 服务** | `localhost:1234/v1` | 文本 → 1024 维向量（bge-m3，OpenAI 兼容接口） | 本地已就绪 ✅（实测返回 1024 维） |
| **DeepSeek API key** | `api.deepseek.com/v1` | LLM 生成答案 | 已有有效 key ✅（不入库，env 注入） |

> 后两项不由 docker-compose 提供，属于外部依赖：bge-m3 由本地 LM Studio 提供、DeepSeek key 通过环境变量注入。
> 二者均已验证可用，每次本地运行前确认 Embedding 服务在线、`DEEPSEEK_API_KEY` 已 export 即可。

## 前置准备

### 1. 启动基础设施（etcd + Qdrant）

```bash
cd /Users/zp/GIT/k8s-deploy/go-zero-rag
docker compose up -d

# 确认两个容器都 Up
docker compose ps

# 确认 Qdrant 就绪
curl http://127.0.0.1:6333/healthz
```

### 2. 启动本地 Embedding 服务（bge-m3）

配置里 Embedding 指向 `http://localhost:1234/v1`，这是 **LM Studio** 的默认端口。
用 LM Studio（或 Ollama / 任意 OpenAI 兼容服务）加载 `bge-m3` 模型并开启本地服务。

> bge-m3 输出 **1024 维**向量，必须与 `etc/*.yaml` 里的 `VectorSize: 1024` 一致；
> 换其它 Embedding 模型时，同步改 yaml 的 `EmbedModel` 与 `VectorSize`。

验证 Embedding 服务可用（应返回 1024 维向量）：

```bash
curl http://localhost:1234/v1/embeddings \
  -H 'Content-Type: application/json' \
  -d '{"model":"bge-m3","input":"hello"}'
```

### 3. 配置 DeepSeek API key

`ai.yaml` 里 `LLMApiKey: ${DEEPSEEK_API_KEY}`，`ai.go` 中的 `conf.UseEnv()` 会从环境变量展开。
启动 aiservice 前必须设置：

```bash
export DEEPSEEK_API_KEY=sk-你的真实key
```

## 启动三个服务

每个服务一个终端（或后台运行）。从仓库根目录执行：

```bash
# 终端 1：doc 服务（文档摄入，:8080）
cd docservice && go run doc.go -f etc/doc.yaml

# 终端 2：ai 服务（RAG 问答，:8082）—— 需先 export DEEPSEEK_API_KEY
cd aiservice && go run ai.go -f etc/ai.yaml

# 终端 3：网关（HTTP 入口，:8888）
cd gateway && go run gateway.go -f etc/gateway.yaml
```

启动顺序无强制要求（etcd 做服务发现，rpc 起来后 gateway 会自动连上），
但建议先起 doc/ai 两个 rpc，再起 gateway。

## 端到端验证

### 1. 上传文档

```bash
curl -X POST localhost:8888/api/doc/upload \
  -H 'Content-Type: application/json' \
  -d '{"title":"go-zero 简介","content":"go-zero 是一个集成了各种工程实践的 web 和 rpc 框架，内置服务发现、负载均衡、限流熔断等微服务治理能力。"}'
```

预期返回（`chunk_count` 为分块数）：

```json
{"doc_id":"...","chunk_count":1}
```

### 2. 提问

```bash
curl -X POST localhost:8888/api/qa/query \
  -H 'Content-Type: application/json' \
  -d '{"question":"go-zero 是什么？","top_k":3}'
```

预期返回（答案 + 命中的来源段落）：

```json
{"answer":"go-zero 是一个 web 和 rpc 框架……","sources":["go-zero 是一个集成了……"]}
```

### 3. 直接查 Qdrant 确认向量已写入

```bash
# 查看 collection（名为 documents）
curl http://127.0.0.1:6333/collections/documents

# 查看已写入的点数量
curl http://127.0.0.1:6333/collections/documents | python3 -c "import sys,json;print('points:',json.load(sys.stdin)['result']['points_count'])"
```

## 常见问题排查

| 现象 | 原因 | 解决 |
|------|------|------|
| gateway 启动报连不上 rpc / etcd | etcd 没起或 doc/ai 没注册成功 | `docker compose ps` 确认 etcd Up；确认两个 rpc 服务已启动无报错 |
| upload 报 embedding 相关错误 | LM Studio/bge-m3 没起，或端口不是 1234 | `curl localhost:1234/v1/embeddings` 自测；改 yaml 的 `EmbedBaseURL` |
| upload 报向量维度不匹配 | Embedding 模型维度 ≠ `VectorSize` | 统一为模型实际维度（bge-m3=1024）；改维度需删旧 collection 重建 |
| query 报 401 / 无答案 | `DEEPSEEK_API_KEY` 未设置或无效 | 确认已 `export`，且启动 ai 服务的终端能读到该变量 |
| 改了维度后检索异常 | 旧 collection 维度不一致 | `curl -X DELETE localhost:6333/collections/documents` 后重新上传 |

## 收尾

```bash
# 停止三个 go 服务：各终端 Ctrl+C

# 停止基础设施（保留数据卷）
docker compose down

# 彻底清理（含 etcd / qdrant 数据卷）
docker compose down -v
```

> 不想依赖云端 LLM 时，把 `ai.yaml` 的 `LLMBaseURL/LLMModel` 也指向本地 OpenAI 兼容服务
> （如 LM Studio 加载的本地对话模型），即可实现完全离线运行。
